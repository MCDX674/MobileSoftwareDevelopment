# Lab14：使用 Room 完成 Bus Schedule 应用

## 一、Entity、DAO、Database 三者在本实验中的职责说明

### 1. Entity（实体）- BusSchedule

**职责**：定义数据模型，映射数据库表结构

- 使用注解标记类，指定对应的数据库表名
- 标记主键字段
- 映射 Kotlin 属性到数据库列名

**在本实验中的作用**：
- 将 Kotlin 数据类映射到 SQLite 数据库中的表
- 定义了公交时刻表的数据结构（编号、站点名称、到站时间）

### 2. DAO（数据访问对象）- BusScheduleDao

**职责**：定义数据库操作接口，封装 SQL 查询语句

- 使用注解标记接口
- 使用注解编写 SQL 查询
- 支持返回 Flow 类型，实现响应式数据更新

**在本实验中的作用**：
- 提供获取完整时刻表的方法
- 提供获取指定站点时刻表的方法
- 封装了 SQL 查询逻辑，ViewModel 无需关心 SQL 语句细节

### 3. Database（数据库）- BusScheduleDatabase

**职责**：创建和管理 Room 数据库实例

- 继承 RoomDatabase 抽象类
- 使用注解声明实体列表和数据库版本
- 提供单例模式的数据库实例

**在本实验中的作用**：
- 创建 Room 数据库实例
- 从 assets 文件夹加载预置数据库
- 提供 DAO 实例的访问入口

## 二、BusSchedule 属性映射关系

### 表名映射
- Kotlin 类名采用驼峰命名法
- 数据库表名采用下划线命名法
- 通过注解指定表名对应关系

### 列名映射
- 站点名称属性对应数据库中的站点名列
- 到站时间属性对应数据库中的到站时间列
- id 属性对应数据库中的主键列

### 为什么需要显式映射？
因为 Kotlin 使用驼峰命名法，而 SQLite 数据库通常使用下划线命名法，需要告诉 Room 如何对应不同的命名风格。

## 三、DAO 查询语句的作用和排序原因

### 第一个查询方法：获取完整时刻表

**作用**：
- 查询数据库表中的所有记录
- 返回完整的公交时刻表数据

**排序原因**：
- 让到站时间从早到晚排列，符合用户阅读习惯
- 用户可以快速找到最早的班车和最晚的班车
- 便于时间规划和行程安排

### 第二个查询方法：获取指定站点的时刻表

**作用**：
- 查询指定站点的所有到站时间
- 参数会被 Room 自动替换，避免 SQL 注入风险

**排序原因**：
- 同一站点的不同班次按时间顺序显示
- 方便乘客查看该站点的班次频率和时间间隔
- 避免时间混乱，便于快速找到目标班次

### 安全机制
使用参数化查询而不是字符串拼接，可以有效防止 SQL 注入攻击。

## 四、createFromAsset() 的作用

### 功能说明

这个方法用于从 assets 目录加载预置的数据库文件。

### 具体作用

1. **预置数据加载**
   - 从 assets 文件夹中的指定路径复制预置的 SQLite 数据库
   - 应用首次启动时自动加载，无需手动插入数据

2. **数据初始化**
   - 保证所有用户有相同的初始数据
   - 避免在代码中编写大量的插入语句

3. **开发和测试便利**
   - 开发者可以预先准备测试数据
   - 便于演示和 QA 测试
   - 数据库结构变更时只需替换文件，无需修改代码

### 文件存放位置
预置数据库文件需要放在 app 模块的 assets 目录下的 database 文件夹中

### 工作流程
1. 应用首次启动
2. Room 检查数据库是否已存在
3. 如果不存在，从 assets 复制预置数据库
4. 如果已存在，直接使用现有数据库

## 五、ViewModel 如何从示例数据切换为数据库数据

### 切换前（使用示例数据）

ViewModel 内部直接创建写死的示例数据列表

通过 flowOf 方法将静态数据包装成 Flow 类型返回

数据无法动态更新，所有用户看到相同的一条示例数据

### 切换后（使用数据库数据）

ViewModel 构造函数接收 DAO 对象作为参数，实现依赖注入

通过调用 DAO 的方法获取数据库查询结果

返回的是 Room 自动生成的 Flow 对象，数据变化时自动更新

### 核心变化

**数据来源变化**：从内部写死的静态数据变为从数据库动态读取

**依赖方式变化**：ViewModel 不再自己创建数据，而是通过构造函数注入 DAO

**响应式支持**：数据库数据支持 Flow，可以自动响应数据变化

**可测试性提升**：通过依赖注入，可以方便地进行单元测试

### 工厂模式的使用

通过伴生对象中的工厂方法创建 ViewModel 实例

工厂负责获取 Application 上下文、创建数据库实例、获取 DAO 对象

将 DAO 传递给 ViewModel 构造函数完成实例化

## 六、Flow 在 Compose 中的收集和显示

### 数据流向

数据库 → DAO（返回 Flow）→ ViewModel（暴露 Flow）→ Compose UI（收集并显示）

### Flow 的特点

- **响应式**：数据变化时自动通知观察者
- **异步**：查询在后台线程执行，不阻塞 UI
- **生命周期感知**：配合 Compose 自动管理订阅和取消订阅

### Compose 收集 Flow 的机制

Compose 提供了专门的函数将 Flow 转换为 State

当 Flow 发射新数据时，State 自动更新

State 变化触发 Composable 函数重组（重新执行）

UI 自动刷新显示最新数据

### 工作流程详解

1. ViewModel 通过 DAO 获取返回 Flow 类型的数据源
2. Composable 函数中使用收集函数订阅 Flow
3. 订阅时会提供一个初始值（通常是空列表）
4. Room 在后台线程执行 SQL 查询
5. 查询结果通过 Flow 发射到收集者
6. State 更新，触发 UI 重组
7. 页面显示数据库中的真实数据

### 优势总结

- **自动更新**：数据变化 UI 自动刷新，无需手动通知
- **线程安全**：Room 自动处理线程切换
- **生命周期安全**：订阅自动跟随 Composable 生命周期
- **性能优化**：只在数据变化时才更新 UI

## 七、实验中遇到的问题与解决过程

### 问题一：Room 编译错误

**现象**：添加 Room 依赖后编译失败，提示找不到符号

**原因分析**：缺少 KSP 编译器插件或依赖配置不完整

**解决方法**：
- 确认在 app 的 build.gradle.kts 中添加了 KSP 插件
- 确认添加了 room-ktx、room-runtime 和 room-compiler 三个依赖
- 执行 Gradle 同步，确保依赖下载完成

### 问题二：运行时提示表不存在

**现象**：应用启动后崩溃，日志显示数据库中没有对应的表

**原因分析**：Entity 中指定的表名与预置数据库中的表名不一致

**解决方法**：
- 检查注解中 tableName 参数的值
- 确保与数据库中的实际表名完全一致（包括大小写）
- 使用正确的方法查看数据库结构进行核对

### 问题三：查询结果为空

**现象**：应用正常运行，但界面不显示任何数据

**原因分析**：DAO 查询语句中的列名与数据库实际列名不匹配

**解决方法**：
- 检查注解中的列名映射是否正确
- 确认参数占位符与变量名一致
- 验证预置数据库文件路径是否正确

### 问题四：项目路径包含非 ASCII 字符

**现象**：Gradle 同步时发出警告，可能导致构建失败

**原因分析**：项目存放在包含中文或特殊字符的路径中，Windows 系统对此支持不佳

**解决方法**：
- 将项目移动到纯英文路径
- 确保路径中只包含字母、数字和下划线
- 重新打开项目并清理重建

### 问题五：ViewModel 实例化失败

**现象**：运行时崩溃，提示无法创建 ViewModel 实例

**原因分析**：未正确配置 ViewModel 的 Factory

**解决方法**：
- 检查 Factory 中获取 Application 上下文的方式
- 确认通过正确的键获取 Application 实例
- 验证数据库创建和 DAO 获取的流程是否正确

### 问题解决总结

- 细心检查注解参数与数据库结构的对应关系
- 确保依赖配置完整并正确同步
- 使用 Logcat 查看详细错误日志
- 通过 Build 视图定位编译错误
- 保持项目路径为纯英文避免平台兼容性问题
